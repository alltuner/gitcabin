A tiny self-hosted GitHub clone driven by gh

Issues, PRs, and counters live in git refs of the bare repo. No database. The official GitHub CLI talks to it natively. Small enough to read in an afternoon. 

# clone, run, point gh at it
git clone https://github.com/alltuner/gitcabin
cd gitcabin
docker compose up -d
echo any-token | gh auth login --hostname github.localhost --with-token
GH_HOST=github.localhost gh issue list

No database

Issues live in refs/issues/*. PRs in refs/prs/*. Counters in refs/meta/*. Code stays in refs/heads/*. Two namespaces, zero collision.

Native gh integration

No new CLI to learn. Set GH_HOST and gh's existing auth, issue, pr, repo commands work. gitcabin serves the GitHub Enterprise URL shape (/api/v3, /api/graphql) plus the github.localhost HTTP shortcut.

Small enough to read

A few thousand lines of Python. FastAPI for REST, Strawberry for GraphQL, GitPython for the storage layer. One container for the API, one for the dashboard. That's it.

Quickstart

Local-only, on your machine, in 60 seconds.

  1. 1. Run the stack

    git clone https://github.com/alltuner/gitcabin
cd gitcabin
docker compose up -d

    Two containers come up: the API on 127.0.0.1:80, the dashboard on 127.0.0.1:8080. Bare repos live under ./data/repos/.

  2. 2. Point gh at it

    echo any-token | gh auth login --hostname github.localhost --with-token
GH_HOST=github.localhost gh auth status

    The token is unverified — gitcabin trusts whoever can reach the port. github.localhost is gh's built-in HTTP-over-port-80 shortcut for local development; nothing else has to be installed.

  3. 3. Use it

    gh issue list --repo octocat/hello
gh issue create --repo octocat/hello --title 'first issue' --body 'hi'
open http://localhost:8080/   # browse the dashboard

Port 80 already in use? See the loopback-alias workaround in the README.

Deployment modes

The local-only quickstart is one of several supported setups. Pick the one that matches your audience.

Mode Audience TLS Setup
Local-only just this machine none (HTTP via github.localhost) docker compose up
Local + TLS just this machine, real cert per-machine local CA one-time admin prompt
Tailnet-shared anyone on your tailnet Tailscale-provisioned add tailscale sidecar
Public / team anyone with DNS resolution Caddy + DNS-01 ACME domain + DNS API token

Full recipes with compose snippets and the matching gh auth login invocation in docs/installation.md.

How it works

Storage in git refs

Each issue is its own ref under refs/issues/<n>. Each comment is a commit on that ref. Counters (next issue number, etc.) live as content-addressed blobs under refs/meta/*. Code is untouched in refs/heads/*.

Plain git clone and git push only see code refs. Metadata is invisible to anyone who hasn't asked for it.

Two URL shapes

gh dials /api/v3/… and /api/graphql for any non-github.localhost hostname (the GHES shape, forced over HTTPS), and bare /, /graphql for github.localhost (the HTTP shortcut). gitcabin serves both, so the same image works behind either deployment mode.

The local-only quickstart uses the HTTP shortcut. TLS deployments use the GHES shape behind a Caddy sidecar.

A few questions

How is this different from gitea / forgejo / gogs?
Smaller. No database. Native gh compatibility instead of a custom CLI. Closer to "what if a GitHub clone fit in your head" than "what if a GitHub clone fit in your data centre."
Is there auth?
Not really. gitcabin trusts whoever can reach the port. Token in gh auth login is accepted unverified. Constrain reach at the network layer (loopback bind, tailnet ACL, reverse proxy basic auth) instead of inventing per-user accounts inside gitcabin.
Can it sync with real GitHub?
Designed for it; not built yet. The authorship and edit-affordance model is in docs/github-sync.md.
Is it production-ready?
No. Pre-1.0. Treat it as a working sketch you can shape into something. Run it on your laptop, file issues, send PRs.
